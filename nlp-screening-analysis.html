System Architecture

High-Level Overview

This is a fully local, privacy-first AI recruitment system that combines a Retrieval-Augmented Generation (RAG) pipeline with a conversational chatbot. It ingests candidate CVs as PDFs, embeds and stores them in a vector database, ranks candidates against a job description, and conducts automated gap-filling interviews via chat. Critically, all LLM inference runs on-device via Ollama — no cloud API calls are made.

⚠ NOTED: The .env file is committed to the repository, exposing the Pinecone API key. This is a critical security issue that must be addressed before any production use (see Improvements section).
Three-Phase Workflow
① Ingest — PDF CVs → chunks → embeddings → Pinecone upsert
② Screen — JD query → similarity search → LLM scoring → gap detection
③ Chat — Select candidate → gap-filling Q&A → HR summary generation
Key Design Decisions
🔒 Local-only LLM via Ollama — no external API calls
📦 Pinecone as managed cloud vector store
🔄 SHA256 IDs for deterministic re-ingestion deduplication
💬 Chainlit for rapid chat UI scaffolding

NLP / RAG Pipeline

Pipeline Pattern: RAG + Conversational AI

The system implements a Retrieval-Augmented Generation (RAG) pattern for candidate screening and a stateful LLM dialogue pattern for the gap-filling interview. The embedding model (nomic-embed-text, 768-dim) handles semantic representation; the generative model (gemma:7b) handles scoring and question generation. Both run locally via Ollama.

01
PDF Ingestion & Text Extraction
PDF CVs from ./cvs/ are parsed to extract raw text. Libraries like PyMuPDF or pdfplumber strip formatting and return plain text per page, handling multi-page documents and encoding issues.
Input: John_Doe.pdf → Output: raw text string Tools: PyMuPDF / pdfplumber (likely) Handles: multi-page, encoded text, whitespace normalization
02
Text Chunking
The extracted CV text is split into overlapping chunks to fit within the embedding model's context window. Each chunk is assigned a deterministic SHA256 ID derived from the candidate ID + chunk index for idempotent Pinecone upserts.
chunk_id = SHA256(f"{candidate_id}_chunk_{i}") Metadata: { candidate_id, source_file, chunk_index, text } Strategy: fixed-size with probable overlap for context continuity
03
Embedding Generation (nomic-embed-text)
Each chunk is sent to Ollama's local nomic-embed-text model to produce a 768-dimensional dense vector. This model is chosen for its strong semantic understanding and MTEB benchmark performance at its size.
Model: nomic-embed-text (via Ollama REST API) Output dimension: 768 Similarity metric: cosine API: POST localhost:11434/api/embeddings
04
Pinecone Upsert (Vector Storage)
Embeddings are upserted to the "recruitment" Pinecone index with full metadata. SHA256 IDs mean re-running ingest.py on the same PDF overwrites existing vectors rather than duplicating them, keeping the index clean.
Index: "recruitment" | Metric: cosine | Dim: 768 Namespace: default (no per-candidate namespacing — see improvements) Upsert: [ { id, values: [768-dim vector], metadata: {...} } ]
05
JD Embedding & Similarity Search
During screening, the job description from config.py is embedded using the same nomic-embed-text model. It is then used to query Pinecone, returning the top-K most semantically similar CV chunks. Results are filtered by candidate_id to aggregate per-candidate context.
jd_vector = embed(JOB_DESCRIPTION) results = pinecone.query(vector=jd_vector, top_k=K, include_metadata=True) Grouped by: metadata["candidate_id"]
06
LLM Scoring & Gap Detection (gemma:7b)
Retrieved CV chunks per candidate are assembled into a prompt alongside the job description and sent to gemma:7b via Ollama. The LLM is instructed to produce a match score (0–100) and identify specific skill/experience gaps. This is the core RAG generation step.
Prompt template (inferred): SYSTEM: "You are an expert recruiter..." USER: "Job Description: {jd}\n\nCandidate CV Excerpts: {chunks}\n\n Score this candidate 0-100 and list skill gaps." Output: { score: int, gaps: [str] }
07
Gap-Filling Conversational Interview
For shortlisted candidates, the chatbot iterates over their detected gaps and generates a targeted interview question for each. The HR user selects a candidate in the Chainlit UI and answers on their behalf (or the candidate answers directly). Answers are captured and stored.
for gap in candidate["gaps"]: question = llm.generate(f"Ask about: {gap}") answer = await chainlit.ask_user(question) candidate["answers"][gap] = answer
08
HR Summary Generation
After all gap questions are answered, the full conversation history — CV chunks, gaps, questions, and answers — is passed to gemma:7b with a summarisation prompt. The model generates a final HR report for the recruiter, which is displayed in the Chainlit UI and persisted to chat_history.json.
Prompt: "Given this candidate's profile and interview answers, write an HR assessment summary..." Output: Markdown/plain-text report Saved to: chat_history.json[candidate_id]["summary"]

Possible Improvements

🔑

Rotate & Remove Exposed API Key

The .env file containing the Pinecone API key is committed to the public repository. The key must be immediately rotated in Pinecone's dashboard and the file must be added to .gitignore. Use environment variables or a secrets manager (e.g. HashiCorp Vault, AWS Secrets Manager) in production.

CRITICAL — DO IMMEDIATELY
🏷️

Use Pinecone Namespaces per Candidate / Job

Currently all candidates are stored in the default Pinecone namespace. Using per-candidate or per-job namespaces would enable cleaner querying (no need to filter by metadata), prevent cross-job contamination, and make deletion/updates trivially scoped without rebuilding the entire index.

HIGH — Architecture Impact
🗄️

Replace JSON Flat Files with a Proper Database

candidates.json and chat_history.json are single-writer flat files that will corrupt under concurrent access and become unwieldy at scale. Replacing them with SQLite (for local use) or PostgreSQL (for multi-user production) would provide transactions, indexing, and proper query capability. SQLAlchemy would make migration easy.

HIGH — Scalability
⚡

Async / Parallel Candidate Ingestion & Screening

Both ingest.py and screening.py likely process candidates sequentially. With asyncio + concurrent.futures or a task queue (Celery/Redis), all candidates could be embedded and scored in parallel. This becomes critical at 50+ CVs where sequential Ollama calls would take many minutes.

MEDIUM — Performance
🔗

Clarify & Properly Integrate the React Frontend

recruitment_ui.jsx constitutes 40% of the codebase but its integration path is unclear — there is no backend API server (FastAPI/Flask) defined in the repository to serve it. It should be either: (a) integrated as a proper React app with a FastAPI backend, or (b) replaced by extending the Chainlit interface with custom elements.

MEDIUM — Architecture Coherence
📊

Add Prompt Engineering & Structured Output Parsing

The LLM scoring step likely produces free-text responses that are parsed heuristically. Switching to structured output prompting (JSON mode via constrained decoding, or Pydantic output parsers via LangChain/LlamaIndex) would make score and gap extraction deterministic, reducing parse failures and prompt sensitivity.

MEDIUM — Reliability
🧪

Add Testing: Unit + Integration

There are no test files in the repository. At minimum: unit tests for the chunking logic and SHA256 ID generation, integration tests for the ingest → Pinecone round-trip (using a local Pinecone mock or a test namespace), and end-to-end tests for the screening pipeline with fixture CVs.

MEDIUM — Code Quality
🤖

Upgrade or Make LLM Model Configurable

gemma:7b is a reasonable default but may produce inconsistent scoring. Consider: (a) adding a config option to swap in llama3, mistral, or phi-3 for comparison, (b) adding a system-level evaluation harness to benchmark different models on a small gold-standard candidate set, and (c) using a larger model (13B+) for the final HR summary generation if hardware permits.

LOW — Capability
📦

Containerise with Docker Compose

Package the entire stack (Python app + Ollama + models) in a docker-compose.yml with an ollama service and an app service. This eliminates the multi-step manual setup (install Ollama, pull 2 models, configure Pinecone, create venv, etc.) and makes the project immediately runnable with a single command anywhere.

LOW — Developer Experience
⚖️

Add Bias Detection & Fairness Guardrails

LLM-based candidate scoring is vulnerable to demographic bias encoded in the training data. The system should include: (a) PII stripping before embedding (names, locations, photos), (b) a bias audit prompt alongside the scoring prompt to flag potentially discriminatory reasoning, and (c) human-in-the-loop confirmation before final shortlisting decisions.

LOW — Ethics
